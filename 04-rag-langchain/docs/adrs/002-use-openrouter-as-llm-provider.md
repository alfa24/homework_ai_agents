# ADR-002: Использование OpenRouter как единого провайдера LLM для MVP

## Статус
Принято

## Дата
2026-05-11

## Контекст

Боту требуется обращаться к LLM для трёх модальностей: текст, изображения (чеки/скриншоты) и аудио (голосовые сообщения). Согласно `docs/vision.md` §6, используется один настраиваемый клиент на базе официального `openai` SDK с переключением провайдера через `base_url`.

Нужно выбрать конкретного провайдера для MVP.

### Рассмотренные альтернативы

1. **OpenRouter** (`https://openrouter.ai/api/v1`)
   - ✅ OpenAI-совместимый API, работает с `openai` SDK без кастомного кода
   - ✅ Единый ключ и биллинг для всех модальностей (текст, vision, audio)
   - ✅ Широкий каталог моделей разных вендоров через один endpoint
   - ✅ Поддержка `whisper-1` для транскрипции голоса
   - ❌ Небольшая наценка поверх прямых API
   - ❌ Зависимость от стороннего агрегатора

2. **Прямые API вендоров** (OpenAI, Anthropic, Google)
   - ✅ Без посредника, нативные фичи
   - ❌ Отдельные аккаунты, ключи, биллинг под каждую модальность
   - ❌ Разные SDK/форматы для vision/audio у разных вендоров
   - ❌ Избыточная сложность для MVP

3. **Локальный Ollama**
   - ✅ Нет стоимости за запросы, полный контроль
   - ❌ Требует железа/настройки, замедляет итерации на этапе проверки идеи
   - ❌ Ограниченная поддержка vision и audio в экосистеме open-source моделей
   - ❌ Противоречит принципу «сначала рабочий end-to-end» (§2)

## Решение

Для MVP используем **OpenRouter** как единого провайдера для всех трёх модальностей.

### Обоснование

1. **Соответствие vision.md §6:** `base_url=https://openrouter.ai/api/v1` подставляется в `AsyncOpenAI`, остальной код не меняется.
2. **Единый контракт на все модальности:** текст, vision и `whisper-1` для аудио — через один клиент.
3. **KISS и минимализм (§2):** один ключ, один провайдер, один биллинг; никаких условных веток по модальностям.
4. **Сохранение пути к Ollama:** паттерн с `base_url` оставляет возможность точечно переключиться на локальные модели в будущем без переписывания слоя клиента.

## Последствия

### Положительные
- ✅ Минимум инфраструктуры для запуска MVP
- ✅ Один `OPENAI_API_KEY` + `OPENAI_BASE_URL` в конфиге (§8)
- ✅ Лёгкая ротация моделей через `MODEL_TEXT` / `MODEL_IMAGE` / `MODEL_AUDIO`
- ✅ Прозрачный путь к voice-итерации (whisper-1 готов «из коробки»)

### Отрицательные
- ⚠️ Наценка OpenRouter поверх прямых цен вендоров
- ⚠️ Зависимость от доступности стороннего сервиса
- ⚠️ Vendor lock-in на уровне биллинга (но не кода — контракт OpenAI-совместимый)

### Миграция
- Переход на прямой OpenAI — смена `base_url` и ключа
- Переход на Ollama — смена `base_url` на локальный endpoint и соответствующих имён моделей
- Изменений в слое сервиса/хендлеров не требуется

## Условия для пересмотра

- Существенный рост расходов, делающий наценку OpenRouter неоправданной
- Требование on-prem/offline работы
- Деградация доступности или качества маршрутизации OpenRouter
- Потребность в фичах конкретного вендора, недоступных через OpenRouter
