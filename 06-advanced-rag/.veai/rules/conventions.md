# Conventions

Полное техническое видение и обоснования — в **`docs/vision.md`**. Не противоречь документу; не пересказывай его в ответах — при необходимости указывай раздел (§).

**Генерация кода**

- Стек и качество: см. `docs/vision.md` §Технологии. RAG-ассистент реализуется на LangChain/LCEL с `InMemoryVectorStore`, semantic/BM25/hybrid retrieval и опциональным cross-encoder reranking; зависимости и провайдеры — строго по vision.
- Объём и стиль: §Принципы разработки — KISS/YAGNI, только нужное для проверки идеи; без преждевременных абстракций и лишних слоёв.
- Размещение и ООП: §Структура проекта — `src/`, `tests/`, `docs/`; **один класс = один файл**; простой OOP.
- Слои и поток: §Архитектура проекта — handler → сервис → LLM/RAG-компоненты; конфигурация и логирование — отдельные инфраструктурные модули.
- Данные: §Модель данных — минимальная модель MVP, без БД; контекст диалога только как ограниченный in-memory буфер при необходимости.
- LLM/RAG: §Работа с LLM и §RAG-ассистент — все модели (`llm`, embeddings, RAGAS embeddings, cross-encoder) и режимы retrieval задаются через env; при сбоях безопасный ответ пользователю, детали только в логах.
- Конфиг: §Подход к конфигурированию — единый `Settings` из окружения; обязательные параметры валидировать при старте.
- Логи: §Подход к логгированию — стандартный `logging`, stdout, без секретов и чувствительных значений.
- Сборка и запуск: не усложняй контракты `make` / uv / Docker без необходимости.
